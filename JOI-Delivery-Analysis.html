Decision

The cart flow deliberately does not handle quantity or decrement stock — /cart/product stays a simple selection list. Reserving stock at cart-add would lock inventory for abandoned carts. The accepted model is to handle quantity + stock at checkout, where it's a real commitment (this is Improvement C, below).

  • inventoryService.reserveStock(items) validates + decrements stock all-or-nothing (it already owns stock)
  • orderService.checkout() prices the lines, builds an Order, returns it
  • New InsufficientStockError → 409; reuse ValidationError (400) / NotFoundError (404)

⚠️ Design caution (applies to any future cart/order work)

Do not populate GroceryStore.inventory Set

Cart responses serialize product.store. If products were added back into the store's inventory Set, you'd create a circular reference:

product → store → inventory → product → store → ...  💥

res.json() would throw on /cart/product and /cart/view. The inventory feature already sidesteps this by deriving products from SeedData.groceryProducts — keep it that way and leave the inventory Set empty.

Security · do first

A · JWT authentication (replace userId in the request)

Problem today

The client sends userId directly in the body/query, so anyone can impersonate another user (Broken Object Level Authorization / IDOR). There is no identity verification — passing userId:"user101" grants access to that user's cart.

Approach
  • Login: POST /auth/login verifies the password (hashed with bcrypt) → issues a short-lived access token (JWT, ~15 min) + a refresh token (~7 days, httpOnly cookie, rotatable)
  • Every authenticated request sends Authorization: Bearer <token>
  • authMiddleware verifies signature + expiry → attaches req.user = { userId, roles } from the claims
  • Controllers/services stop reading userId from the body → use the trusted req.user.userId
  • RBAC: roles customer / store_owner / delivery_partner / admin + a guard middleware (e.g. restrict /inventory/health to that store's owner / admin)
  • Secret in env / KMS with key rotation; use jsonwebtoken
// middleware/auth.js
const jwt = require("jsonwebtoken");
function authenticate(req, res, next) {
  const token = (req.headers.authorization || "").replace("Bearer ", "");
  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET); // { userId, roles }
    next();
  } catch {
    res.status(401).json({ message: "Invalid or expired token" });
  }
}

// app.js — attach middleware
app.post("/cart/product", authenticate, cartController.addProductToCart);

// cartController.js — stop trusting userId from the body
addProductToCart(req, res) {
  const result = cartService.addProductToCartForUser({ ...req.body, userId: req.user.userId });
  ...
}

API impact: POST /cart/product no longer sends userId · GET /cart/view drops ?userId= (derived from the token) · 401/403 become standard responses.

Architecture

B · Split into microservices by bounded context

Problem today

It's a single monolith — cart, product, inventory, and user all live in one process; you can't deploy or scale them independently, and teams collide in the same code. This becomes a bottleneck as it grows.

  • Sync (REST/gRPC) for queries needing an immediate answer (cart asking catalog for a price)
  • Async events (Kafka/RabbitMQ) for state changes to reduce coupling: order emits OrderPlaced → inventory reserves stock, delivery assigns a rider, notification informs the customer
  • Saga pattern for cross-service transactions (order → payment → inventory) with compensating actions when a step fails — instead of distributed transactions
  • Contract testing (Pact) between services to catch breaking changes when deploying independently
Data

C · Real persistence + stock correctness

  • Replace SeedData (in-memory, lost on restart) with a real DB — PostgreSQL (users/stores/products/carts/orders) behind a Repository pattern + a migration tool
  • Stock decrement race condition: an in-memory availableStock -= qty isn't atomic — two concurrent requests can drive stock negative → use an atomic update such as UPDATE ... SET stock = stock - $qty WHERE id = $id AND stock >= $qty (check affected rows) or optimistic locking (a version column)
  • Revisit reserve-on-cart-add: stock is usually reserved at checkout / order creation, not when an item is added to the cart (otherwise abandoned carts lock stock) — consider reservations with a TTL
  • Redis: cache the catalog + hold cart sessions (read-heavy / change-heavy)

A sequenced roadmap

Phase 1 · harden the monolith

Harden

  • JWT auth + RBAC
  • real DB + repositories
  • Joi validation
  • structured logging + /health
Phase 2 · extract services

Extract

  • API Gateway
  • peel off auth / catalog / inventory first
  • DB-per-service
  • contract testing
Phase 3 · event-driven scale

Scale

  • event bus + saga
  • order / delivery / payment
  • real-time tracking
  • tracing + metrics + autoscale

Principle: don't jump straight to microservices — get the bounded contexts clean inside the monolith in Phase 1, then split along the seams once they've stabilized. This avoids paying the distributed-systems tax before you need to.
